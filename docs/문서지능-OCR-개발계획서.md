## 1. 배경과 문제 정의

### 1.1 배경
범용 OCR은 "글자를 읽는 것"까지는 잘 하지만, **문서 전체의 맥락을 이해하지 못한다.** 그 결과 다음과 같은 실패가 반복된다.

- 2단(다단) 레이아웃에서 읽기 순서가 뒤섞여 문단이 깨진다.
- 표가 일반 텍스트로 풀려 셀-행-열 관계가 사라진다.
- 머리글/바닥글/페이지번호/도장·서명 같은 비본문 요소가 본문에 섞인다.
- 여러 페이지에 걸친 표·조항·각주의 연속성이 끊어진다.
- 이미지(도표·차트·다이어그램)가 그냥 "그림"으로 버려지고 의미가 추출되지 않는다.

㈜루다시스템즈는 공공기관 SI(국회 e-의안, 입안지원 시스템 등)와 Local LLM 구축 역량을 보유하고 있어, **"문서를 사람처럼 맥락으로 이해해 정확히 구조화하는"** 추출 엔진을 사내 자산으로 확보할 전략적 가치가 크다.

### 1.2 핵심 문제 (한 줄)
> 스캔/PDF 문서를 받아 **문서 전체 맥락을 파악**하여 **텍스트·이미지·표를 정확히 인지·분리**하고, **읽기 순서와 의미 관계를 보존한** 구조화 데이터(JSON/Markdown/임베딩)로 추출한다.

### 1.3 기존 방식의 한계
| 접근 | 한계 |
|------|------|
| 단순 OCR (Tesseract 등) | 글자만 인식, 레이아웃·표·읽기순서 소실 |
| 클라우드 OCR API 단독 | 정확하지만 데이터 외부 반출, 맥락 보정 약함, 도메인 양식 미세조정 어려움 |
| LLM(VLM) 단독 | 긴 문서/대용량 표에서 hallucination·누락, 좌표 근거 부재, 비용·속도 부담 |

→ **하이브리드**: 결정론적 OCR/레이아웃 엔진으로 좌표·구조를 확보하고, LLM은 맥락 해석·표 정규화·검증에만 투입한다.

## 2. 목표 (Goals)

### 2.1 제품 목표
1. 다양한 한국어 공공·법률·재무 문서를 **사람 수준의 읽기 순서·구조 정확도**로 추출
2. 텍스트 / 표 / 이미지(도표) **3종 요소를 분리 인지**하고 각각에 맞는 표현으로 변환
3. **페이지·섹션을 가로지르는 맥락**(연속 표, 조항 참조, 각주)을 보존
4. 결과를 **구조화 JSON · Markdown · RAG 임베딩** 3종으로 동시 산출
5. **온프레미스 배포 가능**(민감 문서 외부 반출 없음) 구조 확보

### 2.2 PoC 단계 성공 지표 (KPI)
| 지표 | PoC 목표 | 측정 방법 |
|------|----------|-----------|
| 텍스트 인식 정확도 (CER) | ≤ 3% | 골든셋 대비 문자오류율 |
| 읽기 순서 정확도 | ≥ 90% | 문단 순서 일치율 (수동 라벨) |
| 표 구조 복원 (TEDS) | ≥ 0.85 | Tree-Edit-Distance Similarity |
| 표/이미지/텍스트 영역 분류 F1 | ≥ 0.90 | 레이아웃 검출 평가 |
| 필드 추출 정확도(양식류) | ≥ 92% | 키-값 정합 |
| 페이지당 처리 시간 | ≤ 8초 (GPU 1장 기준) | 평균 레이턴시 |

### 2.3 Non-Goals (PoC 범위 제외)
- 손글씨 전용 인식 고도화 (인쇄체·반(半)정형 우선)
- 다국어(영문 외) 전면 지원 — 1차 한국어/영문 혼용까지
- 실시간 스트리밍 처리, 대규모 동시성 튜닝
- 완성형 SaaS 과금·멀티테넌시

| 문서군 | 대표 예시 | 난이도 포인트 |

## 4. 시스템 아키텍처 (하이브리드 파이프라인)

```
 입력 (PDF/이미지/스캔)
        │
        ▼
 ┌──────────────────────────────────────────────────────────┐
 │ 1) 전처리        회전·기울임 보정, 노이즈 제거, 해상도 정규화   │
 ├──────────────────────────────────────────────────────────┤
 │ 2) 레이아웃 분석   텍스트블록/표/그림/머리글·바닥글 영역 검출     │  ← 결정론적 엔진
 │                  + 읽기 순서(reading order) 추정             │
 ├──────────────────────────────────────────────────────────┤
 │ 3) 요소별 인식                                              │
 │    · 텍스트 → OCR (좌표 포함)                                │  ← OCR 엔진
 │    · 표     → 셀 구조 검출 + 셀 OCR                          │
 │    · 이미지 → 캡션/도표 분류, 차트 의미 추출(VLM)              │  ← VLM
 ├──────────────────────────────────────────────────────────┤
 │ 4) 맥락 융합층 (Context Fusion)                             │
 │    · 문서 전역 그래프 구성(섹션·표·각주·참조 연결)             │  ← LLM 후처리
 │    · 페이지 횡단 표/조항 병합, 읽기 순서 교정                 │
 │    · 표 정규화(헤더 추론, 병합셀 해소), 금액·합계 검증         │
 ├──────────────────────────────────────────────────────────┤
 │ 5) 검증·신뢰도   규칙검사 + LLM 자기검증, confidence 스코어     │
 ├──────────────────────────────────────────────────────────┤
 │ 6) 출력 직렬화   구조화 JSON / Markdown / 임베딩(청크화)        │
 └──────────────────────────────────────────────────────────┘
        │
        ▼
 저장소(Document Store) + 벡터DB + 검수 UI
```

### 4.1 왜 하이브리드인가 (역할 분담)
| 단계 | 담당 | 이유 |
|------|------|------|
| 좌표·구조 확보 | 전용 레이아웃/OCR 엔진 | 결정론적, 재현 가능, 좌표 근거 제공 |
| 의미·맥락 해석 | LLM/VLM | 읽기순서·표 헤더 추론·차트 의미화는 추론 필요 |
| 정합성 보증 | 규칙 + LLM 자기검증 | hallucination 억제, 금액/합계 등 수치 검증 |

핵심 원칙: **LLM은 "근거 좌표가 있는 텍스트"만 다룬다.** 원문에 없는 내용 생성을 구조적으로 차단(grounded extraction).

### 4.2 맥락 인지(Context-Aware) 설계 포인트
- **읽기 순서 그래프**: 블록 간 위치·열(column) 관계로 순서를 추정, LLM이 모호한 경우만 교정.
- **문서 전역 메모리**: 섹션 트리(제목 계층), 표 인덱스, 각주/참조 맵을 페이지 처리 시 함께 전달.
- **페이지 횡단 병합**: "표 계속"·"조항 연속" 신호를 감지해 분할된 요소를 하나로 합침.
- **요소 유형 라우팅**: 텍스트/표/이미지에 따라 서로 다른 처리·프롬프트 프로파일 적용.

| 임베딩 | bge-m3 / KURE / OpenAI text-embedding | 한국어 강한 모델 |

## 6. 출력 데이터 모델

### 6.1 구조화 JSON (요소 트리, 발췌)
```json
{
  "doc_id": "uuid",
  "doc_type": "contract",
  "page_count": 12,
  "sections": [
    {
      "id": "s1",
      "title": "제3조 (계약기간)",
      "level": 2,
      "order": 7,
      "blocks": [
        {
          "type": "paragraph",
          "text": "본 계약의 기간은 ...",
          "page": 2,
          "bbox": [120, 340, 980, 410],
          "confidence": 0.97
        }
      ]
    }
  ],
  "tables": [
    {
      "id": "t1",
      "page": [4, 5],
      "spans_pages": true,
      "header": ["항목", "2024", "2025"],
      "rows": [["매출", "120", "150"]],
      "checks": { "row_sum_ok": true },
      "bbox_per_page": { "4": [..], "5": [..] }
    }
  ],
  "figures": [
    {
      "id": "f1", "type": "bar_chart", "page": 6,
      "caption": "연도별 매출",
      "extracted": { "series": [{"label":"2024","value":120}] },
      "bbox": [..], "confidence": 0.88
    }
  ],
  "references": [{ "from": "s1", "to": "t1", "kind": "mentions" }]
}
```

### 6.2 Markdown 재구성
읽기 순서·계층을 보존해 사람이 읽을 수 있는 형태로 재구성한다(제목→`#`, 표→GFM 표, 이미지→`![캡션]` + 추출 수치 주석).

### 6.3 RAG 임베딩 청크화
- 청크 단위: **섹션/표 단위(의미 경계 기준)**, 표는 행 그룹 + 헤더 동봉.
- 메타데이터: `doc_id, page, section_path, element_type, bbox`.
- 각 청크에 **출처 좌표**를 부착해 답변 근거 추적(citation) 가능.

## 7. 개발 단계 / 마일스톤

| 단계 | 핵심 산출물 | 기간(안) |

| P1. 기본 파이프라인 | 전처리+레이아웃+OCR, JSON 직렬화, 검수 뷰어 v0 | 3주 |

총 **약 15주(PoC)**. 단계별 종료 시 KPI 게이트를 통과해야 다음 단계 진입.

## 8. 평가 계획 (검증 가능성)

| 평가 항목 | 메트릭 | 데이터 |
|-----------|--------|--------|
| 텍스트 | CER / WER | 문서군별 골든셋 |
| 레이아웃 | mAP, 읽기순서 일치율 | 영역·순서 라벨 |
| 표 | TEDS, 셀 F1 | 표 구조 라벨 |
| 양식 필드 | 키-값 정확도 | 세무·재무 양식 |
| 임베딩/RAG | Recall@k, 근거 정확도 | 질의셋 |

- **골든셋**: 문서군별 대표 샘플을 수동 라벨링해 회귀 테스트 기준으로 고정.
- **자동 회귀**: CI에서 골든셋 대비 KPI 변동을 추적, 임계 이하 시 빌드 실패.
