## 1. 목적 (Purpose)

본 문서는 PRD에서 정의한 다중 포맷 문서 분석·구조 보존 OCR 시스템을 구현하기 위한 기술적 요구사항, 시스템 아키텍처, 처리 파이프라인, 데이터 명세를 정의한다.

## 2. 시스템 아키텍처 (System Architecture)

시스템은 입력→전처리→포맷 분기→구조 인식→정형 변환→출력의 단방향 파이프라인으로 구성된다.

```
[입력 파일]
    │
    ▼
[1. 파일 수집 / 포맷 식별]  ── MIME·확장자·시그니처 검사
    │
    ▼
[2. 전처리]  ── 이미지 디스큐/노이즈 제거, 텍스트 추출 가능 여부 판정
    │
    ▼
[3. 포맷별 파서 분기]
    ├─ PDF / 스캔 → OCR 엔진 + 레이아웃 분석
    ├─ HWP / DOCX → 스타일·계층 파서
    └─ XLSX → 셀·시트 파서
    │
    ▼
[4. 구조 인식 엔진]  ── 다단 정렬, 표 복원, 캡션/시각요소 태깅
    │
    ▼
[5. 정형 변환기]  ── Markdown / JSON 직렬화
    │
    ▼
[6. 출력 검증 / 반환]  ── 포맷 규격 준수 검사
```

## 3. 구성 요소별 기술 요구사항 (Component Requirements)

### 3.1 포맷 식별 모듈

확장자에 의존하지 않고 파일 시그니처(매직 넘버)와 MIME 타입을 함께 검사하여 포맷을 결정한다. 미지원 포맷은 즉시 오류로 분기하여 파이프라인 자원을 낭비하지 않는다.

### 3.2 전처리 모듈

스캔 이미지의 경우 기울기 보정(deskew), 이진화, 노이즈 제거를 수행한다. PDF는 텍스트 레이어 존재 여부를 판정하여, 텍스트 PDF는 직접 파싱하고 이미지 PDF는 OCR 경로로 보낸다.

### 3.3 포맷별 파서

| 포맷 | 처리 기술 | 핵심 요구사항 |
| :--- | :--- | :--- |
| PDF / 스캔 | OCR 엔진 + 레이아웃 분석 | 다단 영역 검출 후 좌상→우하 읽기 순서로 텍스트 블록 정렬. 그림/그래프 캡션 영역 별도 검출 |
| HWP / DOCX | 문서 객체 모델 파싱 | 개요 번호·글머리 기호의 들여쓰기 레벨 보존, 강조 스타일(bold/underline/italic) 속성 추출 |
| XLSX | 스프레드시트 파싱 | 병합 셀 범위(merge range) 해석, 시트별 분리 처리, 수식 결과값 추출 |

### 3.4 구조 인식 엔진

표 영역을 행·열 그리드로 복원하고, 다단 본문의 논리적 읽기 순서를 재구성하며, 이미지·차트·다이어그램의 위치를 본문 흐름에 맞춰 태깅한다. 제목 계층은 폰트 크기·스타일·번호 체계를 종합하여 `#`~`####` 레벨로 매핑한다.

### 3.5 정형 변환기

인식 결과를 마크다운과 JSON 두 형태로 직렬화한다. 표는 마크다운 표 문법으로, 셀 내 줄바꿈은 `<br>` 태그로 변환한다. 시각 요소는 지정 태그 포맷으로 삽입한다.

## 4. 데이터 명세 (Data Specification)

### 4.1 출력 요소 매핑 규칙

| 원본 요소 | 출력 규칙 |
| :--- | :--- |
| 제목 계층 | `#`, `##`, `###`, `####` 엄격 구분 |
| 본문 텍스트 | 읽기 순서대로 단락 배치 |
| 표 | 마크다운 표, 셀 내 줄바꿈 `<br>` |
| 그림/차트 캡션 | `> [그림 N] 설명` 인용 형식 |
| 시각 요소 | `![[Image/Chart]: 시각 자료 요약]` |
| 개요/글머리 | `-`, 들여쓰기 `    -` 계층 표현 |

### 4.2 표 출력 예시

```markdown
| 구분 | 내용 1 | 내용 2 |
| :--- | :---: | ---: |
| 항목 A | 데이터 A-1<br>(상세내용) | 데이터 A-2 |
```

### 4.3 최종 마크다운 출력 구조

```markdown
# [문서 제목]
*본 문서 분석 결과는 원본의 레이아웃을 기반으로 작성되었습니다.*

---

## [세션/페이지/시트 제목]
[분석된 본문 내용 및 표, 이미지 데이터 배치]
```

별도의 인사말이나 설명 없이 위 구조의 마크다운만 출력한다.

### 4.4 JSON 출력 스키마(개념)

후속 시스템 연계를 위해 요소 단위의 구조화 JSON을 함께 제공한다. 각 요소는 타입(heading/paragraph/table/image/caption), 레벨, 순서 인덱스, 원본 위치(페이지·시트·좌표), 콘텐츠 값을 포함한다.

```json
{
  "document_title": "string",
  "source_format": "pdf|hwp|docx|xlsx",
  "elements": [
    {
      "type": "heading|paragraph|table|image|caption|list",
      "level": 1,
      "order": 0,
      "source": { "page": 1, "sheet": null, "bbox": [x, y, w, h] },
      "content": "..."
    }
  ]
}
```

## 5. 비기능 요구사항 (Non-Functional Requirements)

| 항목 | 요구사항 |
| :--- | :--- |
| 정확도 | 문자 단위 인식 정확도 99% 이상 |
| 출력 규격 | 마크다운/JSON 규격 100% 준수, 검증 단계에서 비규격 출력 차단 |
| 확장성 | 신규 포맷 파서를 플러그인 형태로 추가 가능한 모듈 구조 |
| 무손실성 | 각주·미주·페이지번호 등 부가 정보 누락률 1% 이하 |
| 보안 | 입력 문서는 처리 후 정책에 따라 폐기, 외부 유출 금지 |

## 6. 오류 처리 (Error Handling)

미지원 포맷, 손상 파일, OCR 신뢰도 임계치 미달 시 명확한 오류 코드와 함께 처리를 중단하거나 부분 결과에 신뢰도 경고를 표기한다. 파이프라인 각 단계는 실패 시 후속 단계로 진행하지 않고 즉시 중단·로깅한다.

## 7. 테스트 및 검증 (Testing & Validation)

포맷별 대표 샘플 문서로 회귀 테스트 세트를 구성하고, 원본 대비 텍스트 정확도·표 구조 보존율·읽기 순서 정확도를 자동 측정한다. 출력물은 마크다운/JSON 스키마 유효성 검사를 통과해야 릴리스한다.

## 8. 향후 기술 과제 (Future Technical Work)

PPTX 등 추가 포맷 파서 도입, 요소별 신뢰도 점수 산출, GPU 기반 OCR 가속, 대용량 배치 처리를 위한 병렬 파이프라인 확장을 후속 과제로 둔다.
