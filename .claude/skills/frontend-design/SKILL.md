# Frontend Design

프론트엔드 화면을 설계·구현·리뷰할 때 일관된 디자인 원칙을 적용하기 위한 스킬이다. 새 화면을 만들 때도, 기존 UI를 다듬을 때도 아래 체크리스트를 기준으로 작업한다.

## 1. 시작 전 확인

- 프로젝트에 이미 디자인 토큰/시스템이 있는지 먼저 찾는다(`DESIGN.md`, `tailwind.config`, CSS 변수, 컴포넌트 라이브러리). **있으면 그 토큰을 따르고 임의로 색·간격·라운드 값을 바꾸지 않는다.**
- 없으면 아래 디폴트 토큰 스케일을 제안하고, 사용자 확인 후 적용한다.
- 대상 플랫폼(웹/모바일웹/앱)과 다크모드 지원 여부를 명확히 한다.

## 2. 디자인 토큰 (디폴트 스케일)

기존 토큰이 없을 때만 사용한다.

- **간격(spacing):** 4px 배수 — 4, 8, 12, 16, 24, 32, 48, 64.
- **타이포 스케일:** 12 / 14 / 16(본문) / 20 / 24 / 32 / 48. 줄간격 본문 1.5, 제목 1.2.
- **라운드:** 4(작은 요소) / 8(카드·인풋) / 12(모달) / full(pill·아바타).
- **색:** 중립 grayscale + 브랜드 1색 + 의미색(success/warning/error/info). 색 예산은 페이지당 브랜드색 1~2개로 제한.
- **그림자:** 2~3단계만(sm/md/lg). 남발 금지.

## 3. 레이아웃 원칙

- 8pt 그리드에 정렬. 여백으로 그룹을 구분(테두리·배경보다 여백 우선).
- 콘텐츠 최대폭 제한(본문 ~720px, 대시보드 그리드는 컨테이너 기준).
- 시각적 위계: 크기 → 굵기 → 색 순으로 강조. 한 화면에 강조 1개.
- 반응형: 모바일 우선. breakpoint sm(640)/md(768)/lg(1024)/xl(1280).

## 4. 컴포넌트 상태 (필수 5종)

모든 인터랙티브/데이터 컴포넌트는 다음 상태를 정의한다.

- **default / hover / focus / active / disabled** (인터랙션)
- 데이터 화면은 추가로 **empty / loading / error / success** 를 반드시 처리.
- focus는 키보드 접근성을 위해 항상 보이게(focus-visible ring).

## 5. 접근성 (최소 기준)

- 시맨틱 HTML 우선(`button`, `nav`, `main`, `header`, heading 위계).
- 색 대비 본문 4.5:1, 큰 텍스트 3:1 이상.
- 모든 인터랙티브 요소 키보드 도달 가능 + 포커스 표시.
- 이미지 `alt`, 폼 `label`, 아이콘 버튼 `aria-label`.
- 탭 타깃 최소 44×44px(모바일).

## 6. 마이크로카피 / 모션

- 버튼·에러·빈 상태 문구는 간결하고 행동 지향적으로. 사용자 언어(한국어 입력 시 한국어).
- 모션은 150~250ms, ease-out 위주. 의미 없는 애니메이션 금지. `prefers-reduced-motion` 존중.

## 7. AI 슬롭 회피 (흔한 실수)

- 보라색 그라데이션 남용, 과한 글래스모피즘, 이모지 아이콘 디폴트 → 지양. 아이콘은 SVG 우선.
- 의미 없는 큰 hero 여백, 중앙정렬 일변도, 동일 간격 반복으로 위계 실종 → 점검.
- 그림자·라운드·테두리를 동시에 모든 요소에 적용 → 단순화.

## 8. 작업 마무리 체크리스트

UI 작업을 "완료"라고 말하기 전:

- [ ] 디자인 토큰을 따랐는가(기존 시스템 존중)
- [ ] 5종 상태(+데이터 4종) 모두 처리했는가
- [ ] 키보드/대비/시맨틱 접근성 기준 충족했는가
- [ ] 모바일~데스크톱 반응형 깨짐 없는가
- [ ] 실제로 렌더해 눈으로 확인했는가(스크린샷 또는 브라우저)

> 더 깊은 디자인 생성·리뷰가 필요하면 프로젝트에 있는 `ui-ux-pro-max`, `omd-*`, `design-review` 같은 전용 스킬과 병행한다.
